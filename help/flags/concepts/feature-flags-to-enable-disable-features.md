# 기능을 활성화 및 비활성화하는 기능 플래그 {#feature-flags}

기능 플래그를 사용하면 코드를 재배포하지 않고 런타임에 응용 프로그램 기능을 켜거나 끌 수 있습니다. 또한 코드 배포와 기능 가용성을 구분합니다. 새 코드는 플래그 뒤에 프로덕션에 배포하고 준비가 된 경우에만 켤 수 있습니다.

이 방법은 위험을 크게 줄입니다. 개발자는 민첩성과 신뢰성으로 빌드할 수 있는 반면 제품 팀은 모든 기능 릴리스의 타이밍과 대상을 계속 제어할 수 있습니다.

## 개발 중 종속성 관리 {#manage-dependencies}

기능 플래그는 빠른 개발 사이클에서 테스트할 때 종속성을 관리하는 데 도움이 됩니다. 개발자는 병합 충돌 및 리팩터링을 해결하는 데 더 적은 시간을 소비하고 기능을 제공하는 데 더 많은 시간을 소비합니다.

기능 가용성은 코드 외부에서 제어되므로 복잡한 분기 없이 여러 기능을 동시에 개발할 수 있습니다. 개별 피쳐는 진행 중인 다른 피쳐에 영향을 주지 않고 독립적으로 롤포워드 또는 롤백될 수 있습니다.

## 어두운 배포 {#dark-deployments}

코드 베이스 외부에서 결정을 활성화 및 비활성화하므로 최종 사용자는 기능을 볼 수 없도록 하면서 코드를 프로덕션에 배포할 수 있습니다. 이를 **어두운 배포**&#x200B;라고 합니다.

어두운 배포를 사용하면 코드를 프로덕션에 안전하게 푸시하고, 실제 조건과 비교하여 라이브 환경에서 테스트하며, 배포 일정에 따라 강제로 적용되는 경우가 아니라 선택한 경우 라이브로 전환할 수 있습니다.

## 점진적 롤아웃 {#gradual-rollout}

릴리스할 준비가 되면 기능 플래그를 사용하여 **점진적 롤아웃**&#x200B;을 수행할 수 있습니다. 기능을 사용자 1%로 열고 피드백 및 성능을 측정한 다음 점진적으로 증가시킵니다.

이 순환 접근 방식을 사용하면 제공하는 경험을 보다 세밀하게 제어하고 실제 사용자와의 피드백 루프를 보다 빠르게 제어할 수 있으므로 팀이 전체 릴리스 전에 빠르게 대응할 수 있습니다.

## 중지 스위치 {#kill-switch}

릴리스가 계획대로 수행되지 않으면(비우호적인 피드백, 버그 또는 성능 문제) 기능 플래그를 **종료 스위치**(으)로 사용하여 기능을 즉시 끌 수 있습니다. 코드를 변경하거나 다시 배포할 필요가 없습니다.

## 기능 플래그 라이프사이클 {#lifecycle}

플래그의 기능 플래그는 다음과 같은 일반적인 주기를 따릅니다.

1. 개발자는 기능 플래그를 만들고 다른 사용자에게 노출하지 않고 별도로 테스트합니다.
2. 제품 소유자는 대상을 플래그에 연결하여 정의된 외부 사용자 집합에 기능을 표시합니다.
3. 필요한 경우 관련 플래그와 함께 관리할 [기능 그룹](feature-groups-to-control-multiple-features.md)에 플래그를 추가합니다.
